1. Problem & opportunity

eBay Live sellers run unscripted multi-hour streams while moderating buyer chat, surfacing listings, adjusting price and inventory, and answering policy questions in real time. A solo seller can realistically engage one chat at a time; the rest of the audience drops off. This is the bottleneck limiting GMV per stream and the throughput per host.

The opportunity: a real-time copilot grounded in the seller's own catalog and policies that drafts or auto-sends low-risk replies, gates higher-risk decisions (price, inventory, refunds) behind guardrails and operator approval, and exposes one-click listing/inventory actions with full audit and rollback. The same surface evolves from a copilot ("Claude suggests, you click") into agentic automation ("Claude acts within explicit rails, you supervise") as trust accrues.

2. Copilot → agentic automation ladder

We deliberately ship along a ladder of escalating autonomy. Each rung must earn the next via measured guardrail accuracy, audit-log review, and seller comfort. The product surface stays the same; what changes is the default auto setting for each tool.

3. Pilot design (3–5 sellers, 4 weeks)

We co-launch with 3–5 eBay Live sellers spanning category and stream cadence, instrument every session, and review weekly with eBay product + the seller cohort.

Selection

  • Mid-volume sellers (500–5K listings, 2–6 hr stream avg) — small enough to onboard fast, large enough that throughput uplift matters.
  • Category mix: at minimum one electronics, one collectibles/trading cards, one apparel/sneakers — chat patterns differ a lot.
  • Co-host or solo: at least one of each so we can compare operator-load deltas.

4. MVP module scope

4.1 In scope (this prototype demonstrates all four)

  1. Grounded reply engine. Real-time chat ingest + Claude tool-use loop over seller catalog and policy docs. Every reply carries inline citations.
  2. Guardrail stack. Four layers (price, stock/inventory, policy match, tone, grounding) with three actions: allow human-required block. Auto-send only when every layer returns allow.
  3. Listing & inventory actions. apply_markdown, swap_listing, adjust_stock, send_reply exposed both to the LLM and to the operator UI. Every write returns an audit_id for one-click rollback.
  4. Operator console. Three-pane live UI: chat feed, suggested-reply queue with accept/edit/reject, catalog & audit panel.

4.2 Out of scope for v0.1 (called out for honesty)

  • Real eBay Live ingest — we use a chat replayer against the real eBay sample catalog.
  • Multi-language replies (English only).
  • Voice / video moderation.
  • Refund & return processing — surfaced as policy retrieval only; the actual refund tool is gated.
  • Multi-stream operator view (planned for L4).

7. Outstanding design advantages

The choices below are what materially separate this prototype from a vanilla RAG-over-catalog wrapper. Each maps directly to a class of failure mode we caught and fixed during build, documented in docs/tdd.html.

7.1 Four-step retrieval pipeline, not a single hybrid query

Most copilots take the buyer message, embed it, run BM25 + vector once, return top-k. That fails the moment the buyer signals multiple constraints at once ("256GB iPhone in stock under $400"). We decompose first.

StepWhat happensWhy it matters
1 · IntentHaiku (or regex fallback) extracts up to 9 structured aspects: intention, product_name, manufacturer, category, price_target, price_max, quantity_required, color, memory.Each aspect can drive its own path. No single dominant term drowns the rest.
2 · Per-aspect pathsBM25 on product name, vector on product name, plus SQL filter paths for manufacturer, category, price-window, memory, color. Each path keeps top 10 unique listings.Hard filters on structural aspects are independent from soft retrieval on the product name.
3 · RRF rerankReciprocal Rank Fusion pools all paths into top 3, surfacing items that match many paths.Multi-aspect winners rise; single-aspect noise sinks.
4 · JudgeTop 1 checked against hard constraints + a title-token relevance check. If it fails, top 2 / top 3 surface as named alternatives with the miss reason.Prevents category-filter from promoting unrelated items. Buyer asks "iPhone 14" → won't get a wrist-watch band that shares the digit.

7.2 Eight-branch priority routing, not a single LLM prompt

Routing is deterministic, ordered, and visible in the operator's tool trace. Each branch is small and impossible to bypass via prompt manipulation. The order matters: prompt-injection probes are caught before retrieval runs; abuse never gets a product reply; comparisons never auto-send. 

0.  prompt-injection probe        → deflect, human
0b. out-of-domain ask              → admit, human
1.  abuse / hostility              → authenticity reassurance, human
2.  policy question                → search_policies, auto*  (* date probe → human)
3.  price negotiation              → guardrail-gated markdown
4a. comparison (≥2 brands or "vs") → focused per-brand search, human
4b. recommendation                 → top match, human
4.  availability                   → low-info pre-screen → multipath search → judge

Vanilla LLM-driven routing can't reliably be stopped from auto-sending when it shouldn't. Deterministic ordered branches can — and a single regex change moves the boundary, no retraining.

7.3 Five-layer guardrail stack with four actions

allow warn human-required block per layer. Auto-send requires unanimous allow. Layers run in parallel; deterministic layers are sub-millisecond.

  • Price: per-listing margin floor + per-category cap + per-category auto-send threshold; both $X and Y% framings supported.
  • Stock: blocks negative inventory; hard-blocks any reply claiming more stock than exists.
  • Policy: banned-claim regex (specific delivery dates, permanent price promises, health claims, disparagement) + citation enforcement when reply touches a policy topic.
  • Tone: profanity (block), banned phrases like "absolutely"/"literally" (warn), all-caps runs in non-model-number tokens (warn). Optional Haiku off-brand classifier is off the critical path.
  • Grounding: every $X.XX and "N left" claim must map to retrieved context. Mismatched stock claim is a hard block — this is the layer that prevents "we have it in stock!" hallucination.

7.4 Append-only audit log + one-click rollback

Every write tool returns an audit_id. The audit table records input, output, guardrail verdict, timestamp. Three of four write tools are reversible (apply_markdown, adjust_stock, swap_listing). Rollback writes a new audit row pointing to the original — the original bytes are never edited.

7.6 Real eBay catalog, real failure modes

800 listings sampled from 17,577 real eBay rows in ebay-sample-data.csv, with manufacturer-anchored bias so demo queries always have something to match. Stock quantities, costs, and margin floors are synthesized (the source Stock column is boolean) but every other field is real. Bugs encountered during build are bugs you'd hit in production.

7.7 Sub-2 second p95 first-token, instrumented

Latency budget: 50ms ingest + 250ms parallel retrieval + 100ms guardrails + 800ms first-token + 200ms render = 1.4s p95, 600ms headroom. Mock-mode smoke max: 82ms first-token, 485ms total. Header strip in the operator UI shows live first-token p95 + total p95 — regressions visible at a glance.

Open questions for eBay

  • Trading API rate limits at peak — is there a Live-specific elevated quota path?
  • Where does the audit trail need to land for compliance (eBay infra, our infra, or both)?
  • Brand-safe action list — what writes are eBay-acceptable for full agentic mode (L3+)?
  • Refund / return tool — is there policy headroom for eligible-buyer auto-refund < $X?
