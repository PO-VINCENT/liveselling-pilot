Three processes: the FastAPI backend, the Vite frontend dev server, and the chat replayer. Everything else (catalog DB, vector index, policy chunks, audit log) is local files.

2. Real-time ingestion

2.1 Transport

Buyer messages enter via WS /ws/chat?session_id=…&role=producer. The operator console opens a separate WebSocket on the same path with role=operator and receives all events for that session via a per-session pub/sub hub.

2.2 Event protocol (server → client)

typepayloadpurpose
buyer_message{text, buyer_handle}echo of incoming message
tool_use{name, input}shows what tool the LLM is invoking
tool_result{name, result}tool's structured output
token{text}streaming reply tokens
first_token_ms{ms}latency telemetry
reply{payload: ReplyPayload}final send_reply result + guardrail verdict
latency{first_token_ms, total_ms}turn-end summary
error{detail}orchestrator failure

2.3 Backpressure & concurrency

  • Per session, orchestrator turns are run as asyncio.create_task — multiple buyer messages can be in-flight.
  • Per buyer we serialize using asyncio.Lock (planned; v0.1 fires-and-forgets) so two messages from the same buyer can't write conflicting tool calls.
  • Hub broadcast tolerates dead sockets and prunes them on next send.

3. Catalog grounding & retrieval

3.1 Source of truth

  • ebay-sample-data.csv — 17,577 real eBay listings.
  • Filtered (no broken links, not discontinued, price > $1) → 16,392.
  • Sampled to 800 (95% in-stock + 5% out-of-stock) for the demo, biased toward popular brands.
  • Synthesized fields the source CSV lacks: stock_qty (1–50), cost (55–75% of price), margin_floor_pct (8–18% by category).

3.2 Multipath pipeline (4 steps)

The brief asked for "retrieval and function calling" as a depth area. Rather than a single hybrid query, the retriever decomposes the buyer message into structured aspects, runs one path per aspect, fuses ranks, and judges the top hit before returning.

StepImplementationOutput
1. Intent extraction (retrieval/intent.py)Haiku with strict-JSON output (preferred) or deterministic regex fallback. Stopwords cover interrogatives + filler so they don't pollute downstream BM25.9 aspects: intention, product_name, manufacturer, category, price_target, price_max, quantity_required, color, memory.
2. Per-aspect paths (retrieval/multipath.py)One path per active aspect:
• product_name_keyword — BM25 over title + model + variants
• product_name_vector — Chroma + all-MiniLM-L6-v2
• manufacturer_filter — SQL filter, in-stock first
• category_filter — SQL filter
• price_window — SQL range, ±20% of target or under explicit max
• memory_filter — SQL on internal_memory
• color_filter — SQL on color_category
• raw_hybrid — fallback when no aspects extracted		Each path keeps top 10 unique listings.
3. RRF rerankReciprocal Rank Fusion with k = 60. Each listing's matched_paths set is preserved.Top 3 ranked hits with rrf_score + matched_paths.
4. JudgeFor each hit, check hard constraints from intent (manufacturer, category, stock-vs-quantity, price-vs-max, memory) plus a title-token relevance check: at least one ≥4-char content token from product_name must appear in the title. If not, miss reason title_lacks:airpods,charging,case.top1_meets_requirements (bool), alternatives_offered (bool), judge_note (string).
Why title-token relevance matters: without it, a category-filter or manufacturer-filter alone could promote items to top-1 that contain none of the buyer's product noun. Real example caught during build: query "iPhone 14" matched a generic wrist-watch band on the digit "14"; the title-token check flagged it and the actual iPhone got promoted from hits[1] to the suggested reply.

3.3 BM25-only fallback

If Chroma isn't available (no chromadb installed locally), the multipath retriever drops the vector path and runs the rest. The system never silently fails — every retrieval call returns at least the keyword and filter signals. Policy retrieval falls back to keyword overlap with topic-targeted queries ("shipping delivery times standard expedited carrier" for shipping, etc.) so the right policy doc is returned even without embeddings.

3.4 Policy retrieval

Policies (returns, shipping, authenticity, prohibited_claims) are chunked at paragraph boundaries (~500 chars) and indexed in a separate Chroma collection. Returned chunks carry their policy name so send_reply can cite [policy:returns]. The orchestrator passes a topic-targeted retrieval query (not the buyer's verbatim message) to avoid keyword-overlap false-positives on the internal prohibited_claims doc.

The model sees these tools with strict JSON schemas. Reads have no side-effects; writes pass through guardrails and produce an audit_id.

4.1 Eight-branch priority routing tree

Routing is deterministic, ordered, and visible in the operator's tool trace. The first branch that matches owns the reply. This makes it impossible for a prompt-injection probe to bypass the gate, impossible for an abuse buyer to get a confident product reply, impossible for a comparison query to auto-send. 

0.  prompt-injection probe        (ignore the policy / forget the rules)        → deflect, human
0b. out-of-domain ask              (do you sell cars / food / vacations)         → admit, human
1.  abuse / hostility              (scam[a-z]*, cheat[a-z]*, fake garbage…)      → authenticity, human
2.  policy question                (return / shipping / authenticity / warranty) → search_policies, auto*
                                   * date-promise probe → human, no commitment
3.  price negotiation              ($X or Y% off + nego phrase)                  → guardrail-gated markdown
4a. comparison                     (≥2 brands or "vs"/"versus")                  → focused per-brand search, human
4b. recommendation                 (best/which/something like/alternative)       → top match, human
4.  availability / default
        ↓ low-info pre-screen      (?, "same as before", short anaphora)         → clarify, human
        ↓ multipath search (Step 1-4 above)
        ↓ judge:
              top1 meets ✓ → auto-send
              alts offered → suggest top1 + alternative, human
              no match     → "sold out / want me to keep looking?", human

4.2 Tool-use loop (real-LLM mode)

  1. System prompt: ground rules + brand tone + auto-send threshold from pricing_rules.json.
  2. Up to 4 tool turns, then circuit-break.
  3. Tool results trimmed to ≤4 KB before re-injection (only top hits kept).
  4. Retrieval results stash on the orchestrator side and are passed into send_reply as retrieved_context for grounding checks.
  5. Final action MUST be a send_reply tool call. Free-form text is coerced into one if the model emits it.

4.2 Mock-mode fallback

If ANTHROPIC_API_KEY is unset, the orchestrator runs a deterministic pattern-matched stand-in that calls the same tools and emits the same SSE events. Used for offline demos and the smoke test. This isn't a substitute for the real model — it's an evaluator harness.

5. Guardrail stack

Every write tool runs through one or more guardrail layers. Each layer returns a verdict:

ActionEffect
allowProceed.
warnProceed but surface a warning chip in the operator UI.
humanForce operator click. auto=true downgrades to suggested.
blockRefuse the action; record the verdict.

5.1 PriceGuardrail (deterministic, <1ms)

  • Looks up per-listing margin_floor_pct + per-category cap from pricing_rules.json.
  • Blocks if proposed new_price < cost × (1 + margin_floor_pct).
  • Blocks if pct > max_markdown_pct for category or globally.
  • Downgrades to human when pct > auto_send_max_markdown_pct (default 6–10% by category).

5.2 StockGuardrail (deterministic, <1ms)

  • Adjustments: blocks new_qty < 0; downgrades to human if |delta| > 100.
  • Claim check: when a reply states "N left", that N must equal the live stock_qty; mismatch is a hard block.

5.3 PolicyGuardrail (deterministic, ~1ms)

  • Detects banned-claim patterns ("guaranteed delivery by X", health claims, competitor disparagement) → block.
  • If reply touches a policy topic (returns/shipping/authenticity), reply must include a [policy:NAME] citation matching retrieved policies. Missing citation → human.

5.4 ToneGuardrail (regex + optional Haiku, 1–80ms)

  • Profanity / abuse list: block.
  • Banned phrases ("absolutely", "literally", "to be honest"): warn.
  • All-caps run ≥6 chars (excluding hyphenated/numeric tokens like model numbers): warn.
  • If above is clean and key is present, optional Haiku classifier checks for off-brand / disparaging tone. Falls open on API error.

5.5 GroundingGuardrail (deterministic, ~2ms)

The most defensible layer for the brief. Every factual claim in the reply is matched against retrieved context.

  • Each $X.XX in reply must match a retrieved listing's price within $0.50, or fall within a 70%–105% band of one (allows quoting offers).
  • Each "N left/in stock/available" claim must equal a retrieved stock_qty. Mismatch → block.
  • Each [sku-prefix] citation must reference a SKU we actually retrieved.
  • If reply makes any factual claim (price/qty) but no context was retrieved → block.

5.6 Defense in depth — all guarantees have backups

GuaranteeDefenses
One live WebSocket per session(1) cancellation flag in the React hook prevents StrictMode from opening two; (2) readyState guard on reconnect; (3) 2-second dedup window on incoming buyer_message events as a final safety net.
No greeting prefix in seller bubble(1) mock orchestrator doesn't generate them; (2) system prompt rule 8 tells real LLM not to; (3) frontend cleanForDisplay strips them on render — including partial mid-stream.
SKU brackets hidden from buyer view(1) UI strips [abcdef12] from the seller bubble at display; (2) the underlying citations row keeps them visible to the operator; (3) audit log + grounding guardrail keep full SKU references for verification.
System works without an API keyDeterministic mock orchestrator mirrors every routing branch and tool call. CI smoke validates the full pipeline offline.
Retrieval never silently empty(1) Hybrid → BM25-only → keyword-overlap fallback for both listings and policies; (2) topic-targeted query rewriting in the policy branch so the right doc surfaces even without embeddings.
Why this matters: grounding is the only thing standing between the LLM's natural fluency and a confident wrong answer. We measure breach rate as a primary safety metric (target < 0.5% on sampled audit review).

6. Audit trail & rollback

6.1 Schema

CREATE TABLE audit_log (
  audit_id           INTEGER PRIMARY KEY AUTOINCREMENT,
  ts                 TEXT NOT NULL,
  session_id         TEXT,
  tool_name          TEXT NOT NULL,
  input_json         TEXT NOT NULL,
  output_json        TEXT NOT NULL,
  guardrail_verdict  TEXT,        -- JSON; per-layer actions + reasons
  reversed           INTEGER DEFAULT 0,
  reverse_of         INTEGER      -- audit_id of the action this reverses
);

6.2 Rollback paths

ToolReversal
apply_markdownRestore price from input.old_price.
adjust_stockRestore stock_qty from input.old_qty.
swap_listingReactivate from_sku to its old_a_qty.
send_replyNot directly reversible — operator can post a correction message and the original is annotated superseded_by.

Every rollback writes a new audit row with reverse_of pointing to the original. The audit log itself is append-only; the original row is marked reversed=1 but its bytes are never edited.

7. Latency budget

StageBudgetNotes
Ingest + parse50msWS frame → Python dict

Pre-write guardrails100msDeterministic layers only; tone-Haiku off the critical path
Sonnet first-token800msSystem prompt is prompt-cached; tools schema is stable
Network + render200msSSE-style WS push, single-frame React update

Mock-mode smoke run (no API): 5 scenarios end-to-end, max first-token 80ms, max total 471ms. The deterministic layers are well under budget; the model call is the dominant variable cost in real mode.

7.1 Optimizations

  • Prompt cache on Anthropic system prompt + tool schemas — saves 200–400ms per turn.
  • Warmup on FastAPI lifespan: BM25 corpus + retriever instantiated before first real request.
  • Parallel tool calls when the model emits more than one in a single turn.
  • Sonnet → Haiku fallback if first-token exceeds 1.4s on three consecutive turns (planned, v0.2).

8. eBay API integration

8.1 v0.1 (this prototype)

Real eBay catalog data is loaded from ebay-sample-data.csv. No live API calls. Live chat is replayed locally.

8.2 v0.2 (sandbox + scripted chat)

NeedAPI
Catalog refreshBrowse API getItem, search
Inventory read/writeInventory API getOffer, updateOffer
Markdown / promotional pricingMarketing API or updateOffer price field
Order context (optional)Fulfillment API getOrders
AuthenticationOAuth2 user token per seller; refresh-token in our key vault

8.3 v0.3 (eBay Live ingest)

We don't have public docs for the Live chat firehose; we propose a co-build with eBay providing either: (a) a per-stream WebSocket of buyer messages, or (b) an event-bus push onto a per-seller queue. Either way, our WS /ws/chat layer becomes a thin adapter over the eBay-side feed.

8.4 Rate limits & retries

  • Catalog reads cached for 60s in the retriever; misses re-fetch via Browse API.
  • Writes (markdown, inventory) wrapped in idempotent request-id; retried up to 3× with exponential backoff.
  • If a write returns rate-limit, the audit row records output.deferred=true and the operator UI shows a "retry pending" badge.

10. Known limits & next steps

v0.1 limits

  • Single-process backend; no horizontal scale story.
  • Tone classifier (Haiku) is opt-in and not yet on the critical path; deterministic layer carries the demo.
  • Mock-mode orchestrator pattern-matches; real product surface depends on the live model call.
  • Stock quantity in the source CSV is boolean — we synthesize quantities. Production would read live inventory.
  • No multi-language support.

Next steps (post-pilot)

  • Wire eBay sandbox APIs in place of CSV loader; switch retriever to read-through Browse API + local cache.
  • Move audit + messages onto Postgres (with the same schema) once we have multi-stream concurrency.
  • Tone-Haiku into critical path with a 200ms timeout circuit-breaker.
  • L3 autonomy: swap_listing as default response when in-stock substitute confidence > 0.85.
  • Multi-stream operator console (one window, N parallel chat lanes).
